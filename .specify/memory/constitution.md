# Habit Tracker Constitution

## Core Principles

### I. Disciplined Python Backend (NON-NEGOTIABLE)

Backend MUST be written in Python 3.11+ with full type hints on all function
signatures and return types. All I/O operations MUST be async. Tests MUST be
written before implementation (red → green → refactor); no implementation file
ships without a corresponding failing test written first.

**Rationale**: Course material. Students learn async patterns and TDD from day
one. Violations corrupt the pedagogical sequence and cannot be refactored away
mid-course.

### II. Spec as Single Source of Truth

The `spec.md` for each feature is the single authoritative contract for both
the Python backend and the Bolt-generated React frontend. Spec MUST define all
endpoint paths, HTTP verbs, request/response JSON shapes, and error codes
before any code is written. Neither layer may deviate from the ratified spec.

**Rationale**: Two separate codebases (Python BE + TypeScript FE) cannot share
types. Only a language-agnostic spec prevents silent integration drift.

### III. REST as Language-Agnostic Bridge

All backend ↔ frontend communication MUST use REST/JSON. No backend-specific
transport, no SDK coupling, no GraphQL. AI Coach streaming MUST use
Server-Sent Events (`text/event-stream`) via FastAPI `StreamingResponse`.

**Rationale**: The frontend is generated by Bolt (Vite + React) and cannot
import Python types. REST/JSON is the only portable contract that survives
toolchain changes on either side.

### IV. Vibe Frontend, Spec Contract

The React/TypeScript frontend is generated by Bolt.new — vibe coding is
accepted. However, the spec MUST be passed verbatim as generation context.
Generated code MUST be validated against the spec's acceptance scenarios before
merge. Any frontend deviation from the spec counts as a spec violation, not a
frontend preference.

**Rationale**: Bolt generation is fast but can silently omit edge cases. The
spec is the correctness oracle; validation is non-optional.

### V. Security Boundaries (NON-NEGOTIABLE)

Anthropic API keys MUST reside server-side only — never in frontend env vars,
never bundled in client assets. CORS MUST be `allow_origins=["*"]` in L3 dev
only; L4 production MUST restrict to Lovable/Bolt deployment domains. Supabase
auth (L4) MUST NOT be used to authorize direct Redis access; Redis remains
reachable only through the Python backend.

**Rationale**: API key leakage in public bundles is irreversible once deployed.
CORS restriction is a one-line change deferred to L4 by design — omitting it
in L4 is a violation, not laziness.

## Stack & Technology Constraints

### Backend (L3+)

| Concern | Choice | Constraint |
|---------|--------|-----------|
| Language | Python 3.11+ | Full type hints REQUIRED |
| Framework | FastAPI 0.115+ | async endpoints only |
| Validation | Pydantic v2 | no v1 compat shims |
| Storage | Redis 7 (Docker) | key-value only — NO ORM |
| Testing | pytest + httpx | `FakeAsyncRedis` for unit tests |
| Package manager | uv | no pip-install-direct |

### Frontend (L4+)

- **Generator**: Bolt.new — outputs Vite + React + TypeScript
- **NO Next.js** — Bolt generates Vite; do not migrate the scaffold
- **Data**: habit data stored exclusively in Python backend via REST
- **Auth**: Supabase — auth only, not data storage

### AI Coach (L4+)

- **SDK**: Anthropic Python SDK (server-side)
- **Endpoint**: `POST /coach/advice` → `StreamingResponse` (`text/event-stream`)
- **API key**: server-side env var (`ANTHROPIC_API_KEY`) — never client-side

## Architecture Invariants

Baked into the lesson plan. Do NOT refactor away without a new lesson version.

- **Single-user (L3)**: `userId` hardcoded `"default"`. Multi-user added in L4.
- **Redis key schema** — canonical, no variants, no raw strings outside
  `src/lib/redis_keys.py`:
  - `habits:default:{habitId}` → JSON-serialised Habit object
  - `checks:{habitId}:{YYYY-MM-DD}` → `"1"` (key presence = done, no TTL)
- **Stats on read**: `currentStreak`, `longestStreak`, `successRate`,
  `last30days` computed at request time — NEVER stored.
- **`last30days` index**: `[0]` = today, `[29]` = 30 days ago. Order is strict
  and depended upon by L4 frontend charts.
- **IDs and timestamps**: UUID v4, server-generated. `createdAt` ISO 8601 UTC.
  Client-supplied IDs or timestamps MUST be rejected.
- **Error contract**: 422 Pydantic (auto) · 400 app errors · 404 missing habit
  · 409 duplicate same-day check-in.

## Governance

- Constitution supersedes CLAUDE.md where they conflict. CLAUDE.md provides
  implementation detail that elaborates — not contradicts — the constitution.
- **Amendment procedure**: (1) update rationale in spec, (2) increment version
  per semver rules, (3) run propagation check across all `.specify/templates/`.
- **Version policy**:
  - MAJOR: backward-incompatible principle removal or redefinition
  - MINOR: new principle or material new section added
  - PATCH: clarification, wording, typo fixes
- **Compliance review**: every PR plan.md MUST contain a Constitution Check
  section. Principle violations MUST be entered in the Complexity Tracking
  table with justification — unapproved violations block merge.
- Runtime guidance: see `CLAUDE.md` for detailed implementation rules and
  lesson-scoped constraints.
